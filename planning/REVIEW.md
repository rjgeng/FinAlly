# Review of `planning/PLAN.md`

## Findings

1. High: The LLM execution contract cannot distinguish "suggest a trade" from "execute a trade". [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L300) says the backend auto-executes any returned trades, while [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L371) says the assistant should execute trades only when the user asks or agrees. The current schema only has `message`, `trades`, and `watchlist_changes`, so an analysis-only prompt can still produce a trade payload that gets executed. This needs an explicit execution gate in the contract, such as a separate `intent` field, a `requires_confirmation` flag, or a split between recommendation output and executable actions.

2. High: The plan calls itself the v1 implementation contract, but most API contracts are still underspecified. [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L249) through [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L283) list paths and descriptions, but only `/api/chat` has concrete payload examples. `/api/portfolio`, `/api/watchlist`, `/api/portfolio/history`, `/api/portfolio/trade`, and the SSE stream are missing exact response bodies, error payloads, validation rules, and event names. In a multi-agent build, that is enough ambiguity for the frontend and backend to diverge. The plan should define request and response schemas for every endpoint and every SSE event type.

3. Medium: `portfolio_snapshots` exists, and the frontend depends on it, but the plan never defines when snapshots are recorded. [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L222) creates the table, [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L263) exposes `/api/portfolio/history`, and [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L409) requires a P&L chart from those rows. There is no cadence or trigger defined: on every price tick, on every trade, on an interval, or on page load. That is not a small detail. Different implementations will either spam SQLite writes or leave the chart nearly empty. The plan should lock the snapshot trigger, write interval, and any retention/downsampling rule.

4. Medium: The testing section asks for stable E2E coverage, but the market-data plan remains non-deterministic. The default simulator includes GBM noise plus random events in [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L147) through [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L152), while the test plan only makes the LLM deterministic in [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L500). That leaves price motion, portfolio value, and chart behavior random during tests. For a project that explicitly targets stable E2E runs in [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L535), the plan should define a seeded simulator or dedicated deterministic market mode for test runs.

5. Medium: The plan does not define behavior when a trade or watchlist action references a ticker without a fresh cached price. The UX promises immediate live prices on first launch in [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L15) through [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L20), and immediate market-order fills at the current price in [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L27), but the actual system depends on an in-memory cache populated asynchronously as described in [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L165) through [planning/PLAN.md](/Users/rfvitis/temp/finally/planning/PLAN.md#L178). The contract never states what happens if startup has not populated the default watchlist yet, if a newly added ticker has no simulator seed, or if the Massive-backed price is stale. The plan should define trade rejection rules, freshness thresholds, and the initial-load behavior for uncached tickers.
